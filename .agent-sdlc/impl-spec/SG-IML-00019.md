# Replace UTF-8 URI decoding literals in V1OntologyIndividualController

## Summary

- Problem: `V1OntologyIndividualController` repeats the `"UTF-8"` literal in every `UriUtils.decode` call, which triggers Sonar `java:S1192` and keeps charset selection hardcoded.
- Goal: Replace every controller-local decode literal with `StandardCharsets.UTF_8.name()` and preserve current route behavior.
- In scope:
  - Update every `UriUtils.decode(..., "UTF-8")` call in `backend/src/main/java/uk/ac/ebi/spot/ols/controller/api/v1/V1OntologyIndividualController.java`.
  - Add the `java.nio.charset.StandardCharsets` import if it is not already present.
  - Keep method signatures, endpoint mappings, repository calls, and exception behavior unchanged.
- Out of scope:
  - Any controller outside `V1OntologyIndividualController`.
  - Shared decode helpers or broader URI-decoding refactors.
  - Response payload, status, or query-semantic changes.

## Library And Framework Decisions

- Reuse:
  - `java.nio.charset.StandardCharsets`: Provides the canonical UTF-8 name without a hardcoded string literal.
  - `org.springframework.web.util.UriUtils`: Keep the existing decode API and change only the charset argument source.
- New dependency:
  - `None`: Existing JDK and Spring APIs are sufficient.
- Rejected options:
  - Controller-local `private static final String UTF_8 = "UTF-8"`: Still hardcodes the charset name.
  - Shared decode helper outside this controller: Wider than this backlog item and outside the approved code boundary.

## Functional Requirements

- `FR-1`: Every `UriUtils.decode` call in `V1OntologyIndividualController` must use `StandardCharsets.UTF_8.name()` instead of the `"UTF-8"` literal.
- `FR-2`: The affected routes must keep the same HTTP methods, paths, path-variable names, query parameters, repository lookups, and status behavior.
- `FR-3`: No changes may be made to response field names, exception messages, or non-charset logic.
