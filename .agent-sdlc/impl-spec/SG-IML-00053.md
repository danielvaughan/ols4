# V1SearchController Implementation Spec

## Summary

- Problem: `addCoreFields(...)` contains a long sequence of conditional field-population branches, and Sonar flags its cognitive complexity under `java:S3776`.
- Goal: Reduce the method complexity below the Sonar threshold by extracting cohesive private helpers while preserving the exact field-presence checks, source JSON keys, and output values.
- In scope: Refactor `addCoreFields(...)` and any new private helpers it delegates to within `V1SearchController`.
- Out of scope: Changing the external response contract, rewriting field-name constants, or revisiting the already-completed `search(...)` complexity refactor.

## Phase Plan

### Phase 1: Split addCoreFields into cohesive helpers

- Goal: Lower the method complexity while keeping the mapping logic readable and behaviorally identical.
- Prerequisites:
  - Confirm all in-scope field mappings currently live in `addCoreFields(...)` and are not reused elsewhere.
- Tasks:
  - `P1-T1`: Identify cohesive groups of field mappings, such as identity fields, ontology metadata, textual fields, and score/type/subset fields.
  - `P1-T2`: Extract private helper methods for those groups, for example `addIdentityFields(...)`, `addOntologyFields(...)`, `addTextualFields(...)`, and `addMiscFields(...)`.
  - `P1-T3`: Keep `addCoreFields(...)` as the orchestration method that delegates to the smaller helpers in a stable order.
- Files touched:

## Risks And Operational Notes

- Risk:
  - Behavioral drift during helper extraction, especially around mixed JSON source keys and special-case conversions.
- Mitigation:
  - Extract helpers by cohesive field groups, not by introducing a generic abstraction, and compare each existing branch against the post-refactor mapping.
- Monitoring or rollout note:
  - If manual endpoint checks are performed, compare representative field-filtered search responses before and after the refactor.
