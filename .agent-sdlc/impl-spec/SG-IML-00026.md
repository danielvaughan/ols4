# V1SearchController Implementation Spec

## Summary

- Problem: `V1SearchController` repeats the exact `_json` literal in `3` places inside the Solr field-selection, sorting, and response-document mapping branches inside `search(...)`, which triggers Sonar rule `java:S1192`.
- Goal: Replace those duplicated literal uses with a single controller-scoped constant named `JSON_PAYLOAD_FIELD` without changing the public contract or internal runtime semantics.
- In scope: Add `JSON_PAYLOAD_FIELD` as a private class-level constant and replace the exact standalone `_json` occurrences described by Sonar in `V1SearchController`.
- Out of scope: changing request mappings, renaming public fields or parameters, or merging this change with unrelated refactors, changing `score` field handling (tracked separately by `SG-BLG-00015`) or the broader complexity refactor tracked by `SG-BLG-00009`.

## Library And Framework Decisions

- Reuse:
  - Controller-local `private static final String` constant: smallest maintainable fix for Sonar `java:S1192` in a single controller.
  - Existing controller flow and public field names: must stay unchanged after the constant extraction.
- New dependency:
  - `None`: no new library is required for a local literal extraction.
- Rejected options:
  - Renaming the public field or request parameter `_json`: rejected because the backlog item is only about internal literal deduplication, not contract changes.
  - Folding several neighboring literal extractions into one broad refactor: rejected because each backlog item must remain traceable and independently implementable.
  - Reusing a shared cross-controller constants class: rejected because the requested fix is intentionally controller-local.

- `FR-3`: No exact standalone `_json` literal flagged by this backlog item shall remain in the targeted branch(es) after the refactor.
- `FR-4`: Request mappings, response body shape, Solr parameter semantics, and repository/service interactions shall remain unchanged.
