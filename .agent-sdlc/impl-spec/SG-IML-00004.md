# V1SearchController `exact` Parameter Implementation Spec

## Summary

- Problem: `exact` is declared as `@RequestParam(required = false) boolean exact`, which cannot represent absence and is flagged by Sonar for using a primitive with an optional request parameter.
- Goal: Convert `exact` to a nullable wrapper type and normalize absent values explicitly to today's non-exact-search behavior.
- In scope: The `exact` request parameter declaration and the branches that read it inside `search`.
- Out of scope: Similar fixes for `inclusive` and `isLeaf`, broader refactoring, or changes to the `/api/search` response payload.

## Library And Framework Decisions

- Reuse:
  - `Spring MVC wrapper binding`: `Boolean` keeps the parameter type aligned with the request contract while allowing the absence state.
- New dependency:
  - `None`: the fix is a signature and control-flow cleanup inside the existing controller.
- Rejected options:
  - `Using Object`: loses type safety for a boolean query parameter.
  - `Leaving the primitive in place`: does not satisfy the Sonar finding or make absence explicit.

## Functional Requirements

- `FR-1`: Change the `exact` parameter type from primitive `boolean` to `Boolean`.
- `FR-2`: Normalize the wrapper once near the top of `search` using a local primitive flag such as `Boolean.TRUE.equals(exact)`.
- `FR-3`: Preserve the current behavior where a missing `exact` parameter follows the non-exact search branch and only `exact=true` enables exact-match query construction.
