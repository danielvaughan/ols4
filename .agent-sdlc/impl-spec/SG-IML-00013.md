# V1SearchController `obo_id` Constant Extraction Spec

## Summary

- Problem: `V1SearchController` repeats the `obo_id` response-field key across default field selection and output mapping, producing the duplication tracked by `SG-BLG-00021`.
- Goal: Replace the repeated response-field key with one private constant while preserving the outward `obo_id` field name.
- In scope: The internal `obo_id` key usage in `V1SearchController`.
- Out of scope: Renaming the API field, changing how the CURIE value is sourced, or altering other response-field keys.

## Related Specs

- `Sibling response-field constant extractions in the same controller must preserve the same public field names.`

- `CR-1`: `None.`
- `CR-2`: `None.`

- `backend/src/main/java/uk/ac/ebi/spot/ols/controller/api/v1/V1SearchController.java`: repeats `obo_id` when building the default field list and when checking/adding the response output key.

## Files To Add

- `None.`

## Files To Update

- `backend/src/main/java/uk/ac/ebi/spot/ols/controller/api/v1/V1SearchController.java`: add one private `obo_id` constant and replace the repeated field-key uses.

## Library And Framework Decisions

- Reuse:
  - `Current response-map construction in V1SearchController`: the change is local to field-key reuse.
- New dependency:
  - `None`: no additional library is required.
- Rejected options:
  - `Changing the outward field name`: would break clients expecting `obo_id`.
  - `Keeping the duplicated raw key`: would not satisfy the Sonar finding.

## Functional Requirements

- `FR-1`: Declare one `private static final String` constant for the `obo_id` response-field key.
- `FR-2`: Replace repeated `obo_id` uses in default field-list population and response output mapping with the constant.
- `FR-3`: Preserve current behavior where the response key remains exactly `obo_id` and the value still comes from `curie`.
