# V1OntologyController Implementation Spec

## Summary

- Problem: `V1OntologyController.handleError(HttpServletRequest, Exception)` is an empty public method under `@ExceptionHandler(ResourceNotFoundException.class)` and `@ResponseStatus(HttpStatus.NOT_FOUND)`, which triggers Sonar rule `java:S1186`.
- Goal: Add a nested comment that explains the method is intentionally blank because Spring derives the HTTP 404 response from the surrounding annotations.
- In scope: Update only the body of `handleError(...)` in `V1OntologyController` to include a short explanatory comment.
- Out of scope: Adding logging, changing the response status, throwing `UnsupportedOperationException`, or altering any controller endpoint logic.

## Library And Framework Decisions

- Reuse:
  - Existing Spring MVC exception-handling annotations: they already provide the desired 404 behavior and should remain the only runtime behavior.
  - Inline nested comment: the smallest compliant fix for Sonar `java:S1186` when the empty method is intentional.
- New dependency:
  - `None`: no dependency is required for a comment-only documentation change.
- Rejected options:
  - Throwing `UnsupportedOperationException`: rejected because the exception handler is part of the controller's active 404 flow, not an unsupported code path.
  - Adding logging or side effects: rejected because this backlog item is only about documenting the intentional empty override.
  - Removing the method entirely: rejected because it would change the controller's explicit exception-handler declaration.

- `FR-3`: No logging, no additional exception throwing, and no request/response mutation shall be introduced.
- `FR-4`: The method signature, annotations, and visibility shall remain unchanged.
