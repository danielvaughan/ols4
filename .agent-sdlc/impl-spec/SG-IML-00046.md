# V1SearchController Implementation Spec

## Summary

- Problem: `V1SearchController.applyFilters(...)` currently takes a long list of filter-related arguments, which triggers Sonar rule `java:S107`.
- Goal: Reduce the helper parameter count below the Sonar threshold by encapsulating the filter inputs in a controller-local parameter object without changing caller-visible `/api/search` behavior.
- In scope: Refactor only the internal `applyFilters(...)` helper surface and the immediate call site in `search(...)`.
- Out of scope: Changing request parameter binding, moving logic into another class, or altering any filter semantics.

## Existing Code Touchpoints

- Current filter inputs in `search(...)`: `ontologies`, `slims`, `isLocal`, `isLeaf`, `types`, `groupField`, `childrenOf`, `allChildrenOf`, `inclusive`, and `queryObsoletes` must remain available with the same semantics after the refactor.

## Library And Framework Decisions

- Reuse:
  - Controller-local private nested record or static value type: smallest in-file way to reduce parameter count without widening class boundaries.
  - Existing downstream helper methods such as `addOntologyFilter(...)`, `addSlimFilter(...)`, and `addChildrenFilter(...)`: keep their current responsibilities.
- New dependency:
  - `None`
- Rejected options:
  - Moving filter orchestration into a new service class: rejected because it broadens the backlog beyond a controller-local Sonar fix.
  - Packing the parameters into a raw `Map` or `Object[]`: rejected because it weakens type safety and makes the code harder to maintain.
  - Changing the public `search(...)` request parameters: rejected because the issue is on an internal helper, not the endpoint contract.

## Functional Requirements

- `FR-1`: `applyFilters(...)` shall no longer declare the current long argument list directly.
- `FR-2`: The filter inputs currently passed from `search(...)` shall be represented by a controller-local typed parameter object with one field per existing filter input.
- `FR-3`: All downstream filter helper calls shall continue to receive the same values and preserve the same filter semantics and ordering.
- `FR-4`: The public `search(...)` method signature and `/api/search` request contract shall remain unchanged.
