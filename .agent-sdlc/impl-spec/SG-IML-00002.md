# V1OntologyIndividualController Implementation Spec

## Summary

- Problem: `V1OntologyIndividualController` repeats the exact `"UTF-8"` literal in four `UriUtils.decode(termId, ...)` calls, which triggers SonarQube rule `java:S1192`.
- Goal: Replace those duplicated decode charset literals with one controller-scoped constant without changing any `/api/ontologies/{onto}/individuals/...` behavior.
- In scope: Add one private class-level decode charset constant, import `java.nio.charset.StandardCharsets`, and switch the four existing decode calls in `getIndividual`, `getDirectTypes`, `ancestors`, and `getJsTree` to that constant.
- Out of scope: Changing request mappings, changing error handling, extracting other literals, refactoring repository calls, or introducing broader shared controller utilities.

## Library And Framework Decisions

- Reuse:
  - `java.nio.charset.StandardCharsets`: Preferred over a hardcoded charset literal because it is part of the JDK, stable, and provides the canonical UTF-8 charset name.
  - Controller-local `private static final String` constant: Keeps the Sonar remediation local to the flagged class and avoids widening scope into shared utilities.
- New dependency:
  - `None`: no additional dependency is needed for a single-file literal extraction.
- Rejected options:
  - Repeating `StandardCharsets.UTF_8.name()` inline in each decode call: rejected because it still duplicates the decode charset expression across four methods instead of giving the class one authoritative constant.
  - Creating a shared decode helper in another class: rejected because the backlog scope is intentionally limited to `V1OntologyIndividualController`.
  - Switching to a different `UriUtils.decode` overload without checking framework compatibility: rejected because the current controller already uses the string-based signature and this backlog item is not a Spring API migration.

## Functional Requirements

- `FR-1`: `V1OntologyIndividualController` shall declare exactly one private class-level constant named `URI_DECODE_CHARSET` with value derived from `StandardCharsets.UTF_8.name()`.
- `FR-2`: `getIndividual`, `getDirectTypes`, `ancestors`, and `getJsTree` shall call `UriUtils.decode(termId, URI_DECODE_CHARSET)` instead of passing the `"UTF-8"` literal directly.
- `FR-3`: The refactor shall not change any request path, query parameter, response payload field, repository lookup, or exception type used by these endpoints.
- `FR-4`: After the refactor, `V1OntologyIndividualController.java` shall not contain any standalone `"UTF-8"` literal.
- `FR-5`: New imports required for the refactor shall remain grouped at the top of the file using the repo's standard import ordering.
