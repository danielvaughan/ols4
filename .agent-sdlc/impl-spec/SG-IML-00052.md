# V1SearchController Implementation Spec

## Summary

- Problem: `parseDocumentJson(...)` throws a generic `RuntimeException` when the Solr document is missing the JSON payload field, and Sonar flags that broad exception type under `java:S112`.
- Goal: Replace the generic exception with a more specific unchecked exception, preferably `IllegalStateException`, while preserving the current fail-fast behavior and error-path location.
- In scope: Update only the exception type and message in the missing-payload guard inside `parseDocumentJson(...)`.
- Out of scope: Adding a custom exception hierarchy, changing caller-level handling, or altering payload parsing behavior.

## Existing Code Touchpoints

- Existing `JSON_PAYLOAD_FIELD` contract: indicates this payload is expected to exist before parsing begins, making a state-based exception appropriate.

## Library And Framework Decisions

- Reuse:
  - JDK `IllegalStateException`: appropriate for an internal invariant failure where the expected Solr JSON payload is missing.
  - Existing fail-fast guard location: keep the null check at the earliest point before parsing.
- New dependency:
  - `None`
- Rejected options:
  - Defining a new custom exception type: rejected because the failure is local, internal, and sufficiently described by an existing JDK exception.
  - Silently returning an empty object or fallback payload: rejected because the project prefers fail-fast behavior for invalid internal state.

## Functional Requirements

- `FR-1`: Replace `RuntimeException` with a specific unchecked exception, preferably `IllegalStateException`, in the missing-payload guard.
- `FR-2`: Preserve the fail-fast null check before JSON parsing begins.
- `FR-3`: Keep the exception message descriptive of the missing JSON payload invariant.
