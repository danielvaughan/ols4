# V1SearchController Implementation Spec

## Summary

- Problem: The optional `@RequestParam(value = "isLeaf", required = false)` parameter currently uses primitive `boolean`, which violates Sonar rule `java:S6814` because Spring represents an omitted optional parameter as `null`.
- Goal: Change `isLeaf` to nullable `Boolean` and normalize it inside `search(...)` so the endpoint keeps its current behavior when the parameter is absent.
- In scope: Update only the `isLeaf` request parameter type and the local control-flow checks that depend on it.
- Out of scope: converting `exact` or `inclusive` (tracked separately by `SG-BLG-00010` and `SG-BLG-00011`) or changing leaf-filter semantics.

## Library And Framework Decisions

- Reuse:
  - Wrapper type `Boolean`: compliant object type for an optional Spring MVC request parameter.
  - `Boolean.TRUE.equals(...)` or an equivalent local normalized boolean: keeps the method's runtime semantics explicit when `isLeaf` is omitted.
- New dependency:
  - `None`: the change only uses JDK and Spring types already present in the file.
- Rejected options:
  - Keeping primitive `boolean` with `required = false`: rejected because it is the exact Sonar violation and can map missing input incorrectly.
  - Converting unrelated optional parameters in the same change: rejected because each backlog item is tracked independently.
  - Changing `isLeaf` to a required parameter or adding a default value: rejected because the public request contract must stay optional.

## Functional Requirements

- `FR-1`: The `isLeaf` request parameter in `search(...)` shall change from primitive `boolean` to wrapper `Boolean`.
- `FR-2`: The implementation shall normalize `isLeaf` to the same effective default behavior as today when the request omits the parameter.
- `FR-3`: All existing branches controlled by `isLeaf` shall continue to produce the same Solr query and response behavior for both explicit `true` and missing input.
- `FR-4`: No other request parameter type, default value, or endpoint contract element shall change in this backlog item. Compatible helper extraction or local variable reshaping from `SG-IML-00019` may coexist as long as the caller-visible `isLeaf` behavior remains unchanged.
