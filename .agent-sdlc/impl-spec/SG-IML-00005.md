# V1SearchController `inclusive` Parameter Implementation Spec

## Summary

- Problem: `inclusive` is declared as `@RequestParam(required = false) boolean inclusive`, which is flagged because an optional request parameter should be represented by a nullable object type.
- Goal: Change `inclusive` to `Boolean` and normalize the missing case to `false` so current hierarchy-filter behavior is preserved.
- In scope: The `inclusive` parameter declaration and both hierarchy-filter branches that read it.
- Out of scope: `exact` and `isLeaf` parameter fixes, response mapping, or other `search` refactors.

- `Sibling specs SG-IML-00004 and SG-IML-00006 cover the other optional boolean request parameters in the same method.`

- `CR-2`: `None.`

## Files To Add

- `None.`

## Files To Update

- `backend/src/main/java/uk/ac/ebi/spot/ols/controller/api/v1/V1SearchController.java`: switch `inclusive` to `Boolean`, normalize it once, and route both hierarchy branches through the normalized flag.

## Library And Framework Decisions

- Reuse:
  - `Spring MVC wrapper binding`: `Boolean` keeps type intent clear while allowing omission.
- New dependency:
  - `None`: the fix is fully local to the controller.
- Rejected options:
  - `Using Object`: weakens type safety for a boolean switch.
  - `Changing hierarchy semantics while fixing the signature`: would exceed this backlog item's scope.

## Functional Requirements

- `FR-1`: Change the `inclusive` request parameter type from primitive `boolean` to `Boolean`.
- `FR-2`: Normalize the wrapper once to a local primitive flag such as `Boolean.TRUE.equals(inclusive)`.
- `FR-3`: Preserve current behavior where omitted `inclusive` means only descendants are matched, while `inclusive=true` adds the seed IRIs to the filter set.
