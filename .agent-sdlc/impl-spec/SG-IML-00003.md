# V1SearchController Complexity Reduction Implementation Spec

## Summary

- Problem: `V1SearchController.search(...)` currently mixes parameter normalization, Solr query construction, filter assembly, result mapping, and response building in a single method, driving Sonar cognitive complexity to `89`.
- Goal: Reduce the method below the allowed threshold by extracting cohesive private helpers while keeping request parameters, Solr semantics, response payload keys, and error behavior unchanged.
- In scope: Local refactoring inside `V1SearchController`, including helper extraction for query setup, filter application, default field handling, and response document construction.
- Out of scope: Endpoint contract changes, search semantics changes, new dependencies, schema changes, or edits to other controllers.

- `Sibling specs in this same controller cover optional-boolean parameters and repeated literal extraction; this ticket must not depend on changing the public API shape.`

- `backend/src/main/java/uk/ac/ebi/spot/ols/JsonHelper.java`: supports response document extraction and should remain the single JSON field access utility.

## Library And Framework Decisions

- Reuse:
  - `Spring MVC + SolrJ`: the current controller and `SolrQuery` workflow remain the implementation boundary.
  - `Existing private helpers and utility classes`: reuse `parseFacetFields`, `createUnionQuery`, `SolrFieldMapper`, `JsonHelper`, and `AnnotationExtractor` where they already fit.
- New dependency:
  - `None`: the fix is a local controller refactor.
- Rejected options:
  - `Moving logic into a new service layer for this ticket`: widens scope and changes ownership boundaries beyond the Sonar remediation.
  - `Changing query semantics to simplify the method`: would make the refactor behavior-changing instead of structural.

## Functional Requirements

- `FR-1`: Split `search` into private helper methods grouped by concern, at minimum covering query-mode setup, filter-query application, default field-list initialization, result-document mapping, and response object assembly.
- `FR-2`: Preserve every existing request parameter, default value, branch condition, Solr field name, response key, and `facet_counts` shape.
- `FR-3`: Keep helper scope private to `V1SearchController` unless an already-existing shared utility is a direct fit.
- `FR-4`: Keep all current fail-fast behavior, including the runtime failure when `_json` is unexpectedly absent.

## Phase Plan

### Phase 1: Extract query-construction helpers

- Goal: Remove nested branching from the top half of `search` without changing the generated `SolrQuery`.
- Prerequisites: `None.`
- Tasks:
  - `P1-T1`: Extract a helper for default vs explicit `queryFields` handling.
  - `P1-T2`: Extract helper(s) for filter-query application, including ontology, locality, type, grouping, hierarchy, and obsolete filters.
  - `P1-T3`: Keep the resulting top-level `search` method readable by delegating rather than inlining new branches.
- Files touched:
  - `backend/src/main/java/uk/ac/ebi/spot/ols/controller/api/v1/V1SearchController.java`
- Acceptance criteria:
  - `The top-level query-construction path reads as a sequence of helper calls instead of nested branch blocks.`
  - `The generated Solr query remains equivalent for the current parameter combinations.`

### Phase 2: Extract response-mapping helpers

- Goal: Reduce complexity in the result-processing half of `search`.
- Prerequisites: `Phase 1 complete.`
- Tasks:
  - `P2-T1`: Extract a helper that initializes the default `fieldList` when the request does not supply one.
  - `P2-T2`: Extract a helper that maps a single `SolrDocument` into the output map while preserving existing keys and annotation behavior.
  - `P2-T3`: Extract final response-object assembly so the main method only orchestrates request, query, and response stages.
- Files touched:
  - `backend/src/main/java/uk/ac/ebi/spot/ols/controller/api/v1/V1SearchController.java`
- Acceptance criteria:
  - The main `search` method no longer owns the detailed field-mapping and response-object construction logic.
  - Existing keys such as `responseHeader`, `response`, `docs`, and `facet_counts` remain unchanged.

## Risks And Operational Notes

- Risk:
  - `Helper extraction can accidentally change query ordering, filter composition, or output keys if the refactor is not kept mechanically equivalent.`
- Mitigation:

- Monitoring or rollout note:
  - `Because this endpoint is public and heavily parameterized, implementation review should compare generated query fragments and response keys before merge.`
