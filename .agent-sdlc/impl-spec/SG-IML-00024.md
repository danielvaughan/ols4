# V1SearchController Implementation Spec

## Summary

- Problem: `V1SearchController` repeats the exact `edismax` literal in `3` places inside the Solr query parser setup inside `search(...)`, which triggers Sonar rule `java:S1192`.
- Goal: Replace those duplicated literal uses with a single controller-scoped constant named `EDISMAX_QUERY_PARSER` without changing the public contract or internal runtime semantics.
- In scope: Add `EDISMAX_QUERY_PARSER` as a private class-level constant and replace the exact standalone `edismax` occurrences described by Sonar in `V1SearchController`.
- Out of scope: changing request mappings, renaming public fields or parameters, or merging this change with unrelated refactors, changing the `"defType"` parameter name (tracked separately by `SG-BLG-00013`) or the broader complexity refactor tracked by `SG-BLG-00009`.

## Library And Framework Decisions

- Reuse:
  - Controller-local `private static final String` constant: smallest maintainable fix for Sonar `java:S1192` in a single controller.
  - Existing controller flow and public field names: must stay unchanged after the constant extraction.
- New dependency:
  - `None`: no new library is required for a local literal extraction.
- Rejected options:
  - Renaming the public field or request parameter `edismax`: rejected because the backlog item is only about internal literal deduplication, not contract changes.
  - Folding several neighboring literal extractions into one broad refactor: rejected because each backlog item must remain traceable and independently implementable.
  - Reusing a shared cross-controller constants class: rejected because the requested fix is intentionally controller-local.
